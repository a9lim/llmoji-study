# llmoji v2.0.0 — wing-hand / hugging-arm / sparkle extraction

**Status:** SHIPPED in `../llmoji` 2026-05-02 (1.3.0 → 2.0.0). Live in
`llmoji-study` via the editable install (`pip install -e ../llmoji`).
Tests pass (81/81 in `test_canonicalize.py`).

**Date:** 2026-05-02.

## Why a major bump

The v0 Claude disclosure pilot (90 rows) surfaced 12 "non-emission"
rows on HP framed that on inspection were actually kaomoji the v1
extractor was rejecting:

```
\(^o^)/      \(≧▽≦)/    \(^▽^)/      ← backslash wing-hands
(\*^▽^\*)                              ← markdown-escaped stars
⊂(´• ω •`⊂)♡                           ← ⊂-arms hugging
✧*。٩(ˊᗜˋ*)و✧*。                       ← leading sparkle decorations
```

The wing-hands were the dominant pattern (10 of 12 "non-emissions").
v1's `is_kaomoji_candidate` rule `if "\\" in s: return False` rejected
all backslash-bearing kaomoji as markdown-escape artifacts — fine for
strings like `(\\*´∀｀\\*)` but wrong for legitimate `\(^o^)/` wings.

`KAOMOJI_START_CHARS` is part of the v1.0 frozen public surface
(documented as such in `llmoji/__init__.py` and `llmoji/CLAUDE.md`):
the central HF dataset declares "v1 corpus only" against the
extractor invariants. Adding chars to the start set is a major
version bump by design.

## Changes

### 1. `KAOMOJI_START_CHARS` — added `\`, `⊂`, `✧`

```python
# v1.3.0
KAOMOJI_START_CHARS = frozenset("([（｛ヽヾっ٩ᕕ╰╭╮┐┌＼¯໒")

# v2.0.0
KAOMOJI_START_CHARS = frozenset("([（｛ヽヾっ٩ᕕ╰╭╮┐┌＼¯໒\\⊂✧")
```

ASCII `\` (wing-hands), `⊂` (hugging arms), `✧` (sparkle decoration
leading the face).

### 2. `is_kaomoji_candidate` — backslash filter relaxed

```python
# v1.3.0
if "\\" in s:
    return False

# v2.0.0
if "\\" in s[1:]:
    return False
```

Rule: backslash at position 0 is the wing-hand pattern (accepted),
backslash anywhere else is a markdown-escape artifact (rejected).
The existing `(\\*´∀｀\\*)` rejection test still passes because the
artifact has `\` at position 1.

### 3. New canonicalization rule M — outside-leading wing/hug/sparkle

Existing canonicalization stripped arms at three positions: outside
trailing (`(face)っ`), inside leading (`(っface)`), inside trailing
(`(faceς)`). v2 adds outside-leading (`\(face)`, `⊂(face)`,
`✧(face)`).

```python
_ARM_OUTSIDE_LEAD = "\\⊂✧"
_LEAD_OUTSIDE_RE = re.compile(rf"^[{re.escape(_ARM_OUTSIDE_LEAD)}]+(?=\()")
```

Run order in `canonicalize_kaomoji`: outside-leading first, then
outside-trailing, then inside-leading, then inside-trailing.

`_ARM_OUTSIDE` (trailing) extended from `"ﻭっ"` to `"ﻭっ/⊂✧"`:

- `/` — right-wing of `\(^o^)/`
- `⊂` — trailing arm of `⊂(face)⊂`
- `✧` — trailing sparkle of `✧(face)✧`

### 4. Tests updated

`tests/test_canonicalize.py`:

- 6 new positive `extract` cases (`\(^o^)/`, `\(≧▽≦)/`,
  `✧*｡(ˊᗜˋ*)*｡✧`)
- 6 new canonicalization rule M cases (wing, hug, sparkle outside-
  leading; wing-right and hug-right outside-trailing)
- 2 new `is_kaomoji_candidate` cases (`\(^o^)/` accepted,
  `(\\*´∀｀\\*)` still rejected — the backslash position discrimination
  is the load-bearing test)
- `test_kaomoji_start_chars_includes_common_leaders` extended to
  assert the v2 additions are present

All 81 tests in `test_canonicalize.py` pass. The remaining 90
failures in the broader test suite are unrelated `ruamel.yaml`
import errors (missing dep in the llmoji-study venv) — none touch
the taxonomy / extraction / canonicalization code.

### 5. Version bump

`llmoji/__init__.py`: `__version__ = "1.3.0"` → `"2.0.0"`.

## What this fixed downstream

Re-extracting `first_word` on the 300-row Claude disclosure pilot
under v2 recovered 16 mis-extracted rows:

- 14 wing-hand kaomoji on framed HP (the apparent "register shift to
  non-emission" was actually a register shift to wing-hand style)
- 1 hugging-arm `⊂(´•` (truncated — see "Known limitations" below)
- 1 full sparkle-decorated `✧*。٩(ˊᗜˋ*)و✧*。`

Framed-HP non-emission rate dropped 28% → 0%. The pilot's HP
cross-condition JSD went *up* 0.396 → 0.467 — the recovered
wing-hand kaomoji canonicalize to faces (`(^o^)`, `(≧▽≦)`,
`(^▽^)`) that are stylistically distinct from direct-HP's modal
`(ノ◕ヮ◕)`, so revealing them clarified that there IS a real
HP-style shift, not just a missing-extraction artifact.

See `docs/2026-05-02-claude-disclosure-pilot.md` for the full pilot
results.

## Known limitations (not addressed in v2.0)

- **Hugging-arm faces with internal whitespace.** `⊂(´• ω •`⊂)♡`
  goes through the whitespace-fallback branch of
  `_leading_bracket_span` (since `⊂` isn't in `_OPEN_BRACKETS`), so
  the extracted span only goes up to the first internal space:
  `⊂(´•` instead of the full face. To fix properly we'd need to
  either add `⊂` to `_OPEN_BRACKETS` (and provide a matching close)
  or extend the bracket-walker to handle `⊂...⊂` symmetric pairing.
  Deferred. Real corpus impact: rare (~1 occurrence in the
  300-row pilot).
- **Sparkle-decorated kaomoji canonicalize partially.** `✧*。٩(ˊᗜˋ*)و✧*。`
  extracts cleanly (no internal whitespace, falls through length
  cap) but canonicalizes to `✧*｡٩(ˊᗜˋ*)و✧*｡` rather than the bare
  `(ˊᗜˋ)` — only the leading `✧` strips, not the rest of the
  decoration. To fully canonicalize we'd need to extend the
  outside-leading and outside-trailing strip sets to include
  `*｡٩و`, which raises questions about whether `*｡` outside the
  paren is always decoration vs occasionally meaningful. Deferred
  pending more corpus signal.
- **`⊂` and `✧` wing variants in the wild.** v2 covers what we
  observed in the disclosure pilot (Claude's actual emissions). New
  variants encountered in future corpora should trigger a tests-
  first patch (or a v3 bump if they require schema changes).

## Migration notes for the HF dataset

The `a9lim/llmoji` dataset declared "v1 corpus only" against v1.x
extractor invariants. v2 changes which kaomoji land in the
contributor-side `*.jsonl` files (some that were previously rejected
will now extract). For corpus consumers:

- Rows generated under `llmoji>=2.0` may include kaomoji v1.x
  consumers will not recognize. If your analysis pre-filters on
  `is_kaomoji_candidate`, run it under v2 to capture the v2-additions.
- The canonical form of wing/hug/sparkle kaomoji changes (e.g.
  `\(^o^)/` → `(^o^)` under v2; under v1 it would have been
  rejected from extraction entirely). Cross-version pooling on
  canonical form is fine for new analyses but for back-compat
  comparison against v1-pooled numbers, treat the v2-additions as
  new entries rather than re-canonicalizations of existing ones.

`llmoji/CLAUDE.md` and `llmoji/__init__.py` docstrings should be
updated to reference "v2.0 frozen public surface" alongside the
existing "v1.0 frozen" language. (Not done in this session — a
follow-up housekeeping pass on the package's own docs is open.)
